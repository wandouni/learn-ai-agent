① MCP 是什么——先建立直觉

🔌 比喻：MCP 就像 USB 接口标准。
你的电脑（Claude）只要有 USB 口，就能接任意厂商的设备（数据库、文件系统、API）。设备按统一规范制造，电脑不需要为每个设备单独写驱动。

MCP（Model Context Protocol）是 Anthropic 定义的协议，解决一个问题：大模型怎么安全、统一地调用外部工具和数据？

Client 客户端

Claude Desktop
向服务端发请求、接收结果

Server 服务端

你写的 server.py
暴露工具、执行操作

Transport 传输层

stdio（标准输入输出）
两者通过管道通信，不需要 HTTP

② 整体架构——谁在和谁说话

你（用户）
在 Claude Desktop 输入自然语言
↕
HTTP / WebSocket（Claude Desktop 内部）
☁️ Claude 大模型（云端）
理解自然语言 · 决定是否调用工具 · 组织最终回答
↔
Claude Desktop
本地 App
协议中转站
↕
stdio（标准输入输出管道）—— JSON-RPC 消息
🐍 server.py（本地进程）
注册三个工具 · 执行 SQL · 返回结果
↕
Python sqlite3 库
🗄️ demo.db（本地文件）
employees 表 · departments 表
关键点：Claude 大模型在云端，它看不见你的数据库文件。 它只能通过"工具调用"请求本地的 server.py 去查询，再把结果返回给它。 数据不会直接上传云端，server.py 是数据的守门人。

③ 代码逐段解析——每行在做什么

第一部分：初始化

from mcp.server.fastmcp import FastMCP DB_PATH = os.path.join(os.path.dirname(__file__), "demo.db") mcp = FastMCP("SQLite 查询助手")
FastMCP 是 MCP SDK 提供的高级封装。创建实例时传入服务名称，这个名称会在 Claude Desktop 的工具列表里显示。 DB_PATH 固定指向 server.py 同目录下的 demo.db，避免路径问题。

第二部分：注册工具（最核心的部分）

@mcp.tool() def list_tables() -> str: """列出数据库中所有的表名。""" ...
装饰器 @mcp.tool() 做了三件事：
1. 把这个函数登记到 MCP 工具列表
2. 从函数签名自动推断参数的 JSON Schema（Claude 需要这个来知道怎么调用）
3. 把 docstring 作为工具描述发给 Claude（Claude 靠这段描述决定"要不要调这个工具"）

三个工具的职责对比

工具入参做什么对应 SQL
list_tables无查 sqlite_master 系统表，列出所有用户表SELECT name FROM sqlite_master WHERE type='table'
describe_tabletable_name用 PRAGMA 获取字段元信息PRAGMA table_info(表名)
run_querysql执行任意 SELECT，先校验再执行用户提供的 SQL

第三部分：启动入口

if __name__ == "__main__": init_db() # 数据库不存在就创建示例数据 mcp.run(transport="stdio") # 进入监听循环，通过 stdin/stdout 收发消息
mcp.run(transport="stdio") 启动后程序不会退出，它进入一个事件循环，持续监听 stdin 上的 JSON-RPC 请求，处理后把结果写到 stdout。Claude Desktop 把这个进程的 stdout/stdin 接管，充当"管道"。

④ 完整交互流程——以"查薪资最高5人"为例

1
用户 → Claude Desktop
输入自然语言

「帮我查薪资最高的 5 个员工」

2
Claude Desktop → 云端大模型
把用户消息 + 工具列表一起发给 Claude

大模型收到的不只是用户的问题，还附带了"现在有哪些工具可以用"的清单（由 server.py 的工具注册信息生成）：

{ "tools": [ { "name": "list_tables", "description": "列出数据库中所有的表名" }, { "name": "describe_table", "description": "查看指定表的字段结构", "inputSchema": { "table_name": "string" } }, { "name": "run_query", "description": "执行 SELECT 查询", "inputSchema": { "sql": "string" } } ], "messages": [{ "role": "user", "content": "帮我查薪资最高的 5 个员工" }] }
3
云端大模型（思考中）
模型推理：我需要先知道表结构，才能写 SQL

模型内部推断：要查员工，先得知道 employees 表有哪些字段。决定先调 describe_table。

模型返回一个"工具调用请求"（不是文字答案）：

{ "type": "tool_use", "name": "describe_table", "input": { "table_name": "employees" } }
4
Claude Desktop → server.py（通过 stdio）
转发工具调用请求给本地进程

Claude Desktop 收到大模型的工具调用请求后，通过 stdin 把 JSON-RPC 消息写给 server.py：

{ "jsonrpc": "2.0", "method": "tools/call", "params": { "name": "describe_table", "arguments": { "table_name": "employees" } } }
5
server.py 执行
describe_table("employees") 被调用，查询 SQLite

server.py 执行 PRAGMA table_info(employees)，把结果格式化成字符串，写回 stdout：

{ "result": { "content": [{ "type": "text", "text": "字段名 | 类型 | 主键\nid | INTEGER | 是\nname | TEXT | 否\nsalary | REAL | 否\n..." }] } }
6
Claude Desktop → 云端大模型（第二轮）
把工具结果追加进对话，再次发给模型

现在对话历史变成了：用户提问 → 模型发出工具调用 → 工具返回结果。大模型拿到表结构后，继续推理并决定调用 run_query：

{ "type": "tool_use", "name": "run_query", "input": { "sql": "SELECT name, department, salary FROM employees ORDER BY salary DESC LIMIT 5" } }
7
server.py 执行
run_query 检查语句 → 执行 → 返回结果

先验证是否以 SELECT 开头（通过），然后查询 SQLite 返回 5 行数据。

8
云端大模型 → 用户
模型把数据整理成自然语言回复

拿到查询结果后，模型不再发出工具调用，而是直接生成最终回答，用自然语言描述薪资排名。

⑤ stdio 消息格式——管道里流动的是什么

Claude Desktop 和 server.py 之间通过操作系统的"管道"通信。本质上就是两个进程共享 stdin/stdout，数据格式是 JSON-RPC 2.0，每条消息是一行 JSON。
→ 请求 Desktop 发给 server.py
{ "jsonrpc": "2.0", "id": 1, "method": "tools/call", "params": { "name": "run_query", "arguments": { "sql": "SELECT ..." } } }
← 响应 server.py 返回给 Desktop
{ "jsonrpc": "2.0", "id": 1, "result": { "content": [{ "type": "text", "text": "name | salary\n..." }], "isError": false } }

启动时还有握手阶段：Desktop 先发 initialize 请求，server.py 返回自己支持的能力和工具列表。之后才开始正常的工具调用。

⑥ 安全拦截——为什么删库命令不会执行

✅ 允许
  • SELECT * FROM employees
  • SELECT name FROM departments
  • SELECT ... JOIN ... WHERE ...
🚫 拒绝
  • DELETE FROM employees
  • DROP TABLE employees
  • INSERT INTO ...
  • UPDATE employees SET ...
拦截逻辑在 server.py 里，发生在数据库执行之前：
if not stripped.upper().startswith("SELECT"): return "拒绝执行..."

即使 Claude 大模型"想"执行删除操作，server.py 在本地就挡掉了，不会触达 SQLite。 大模型只能得到一条错误文本，它会把这个拒绝原因转述给用户。

⑦ 一句话总结每个角色

角色位置职责
你（用户） 人 用自然语言提问，看最终回答
Claude 大模型 云端 理解意图、推理要调哪个工具、生成 SQL、组织最终回答
Claude Desktop 本地 App 中转站：把大模型的工具调用请求转给本地进程，把结果转回给大模型
server.py 本地进程 守门人：注册工具、安全校验、执行 SQL、格式化结果
demo.db 本地文件 数据源：存储真实数据，只接受 server.py 的访问
stdio 管道 操作系统 传输层：Desktop 和 server.py 之间的 JSON-RPC 通信通道
💡 最简理解：你跟 Claude 说话 → Claude 大脑决定要查数据库 → 发出"工具调用"请求 → Claude Desktop 把请求转给你本地的 server.py → server.py 查 SQLite → 结果原路返回 → Claude 大脑拿到数据后用自然语言告诉你答案。
整个过程你的数据库数据不会上传云端。

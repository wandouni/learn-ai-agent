## 项目定义

**名称：** Excel 报表 AI 分析自动化

**一句话目标：** 定时扫描指定目录下的 Excel 文件，自动提取关键数据，调用 DeepSeek 生成分析摘要，输出 Markdown 报告，并可选推送到钉钉或邮件。

---

## 背景：解决什么问题

很多公司的数据汇报流程是这样的：各部门每周填一份 Excel，发到一个共享目录或群里，然后有人负责把这些表汇总、写成一段分析发给领导。这个过程重复、低价值、耗时。

这个项目模拟的就是把这个过程自动化： **程序自动读表 → AI 自动写分析 → 自动推送给相关人** 。

---

## 系统边界

```
定时任务（每天定时触发）
     ↓
扫描 Excel 目录（多个部门的报表文件）
     ↓
数据提取与结构化
     ↓
构造 Prompt → DeepSeek API → 分析摘要
     ↓
生成 Markdown 报告文件
     ↓（可选）
推送到钉钉群 / 发送邮件
```

---

## 输入：Excel 文件规范

为了让练习可控，约定所有 Excel 文件遵循统一格式，模拟"各部门按模板填报"的场景。

每份 Excel 文件代表一个部门的周报，包含以下字段：

* 部门名称
* 汇报周期（如 2025-W23）
* 本周完成事项（文字，若干条）
* 本周关键指标（表格，指标名 + 数值 + 环比变化）
* 下周计划（文字，若干条）
* 风险与问题（文字，可为空）

目录下可能有 1 到 N 个文件，文件名格式为 `部门名_周报_日期.xlsx`，程序每次运行处理当天或本周的所有文件。

---

## 数据提取逻辑

读取每个 Excel 文件后，提取以下结构化信息：

* 部门名称、汇报周期
* 关键指标列表：每条包含指标名、数值、环比（涨跌方向 + 百分比）
* 完成事项列表
* 下周计划列表
* 风险问题列表（如有）

提取完成后做基础校验：

* 必填字段是否为空
* 指标数值是否为数字
* 文件格式是否符合模板

校验不通过的文件记录错误日志，跳过处理，不中断整体流程。

---

## 两种分析模式

### 模式一：单部门分析

对每个部门的 Excel 单独生成一段分析，重点回答：

* 本周整体表现如何，关键指标趋势是好是坏
* 完成事项与计划的匹配度
* 风险问题的严重程度判断
* 下周计划是否合理、有无遗漏

### 模式二：跨部门汇总分析

在所有单部门分析完成后，把所有部门的结构化数据合并，再调用一次 DeepSeek，生成一份整体分析，重点回答：

* 哪个部门本周表现最好 / 需要关注
* 各部门指标的横向对比
* 全公司层面的共性风险或亮点
* 给管理层的 2-3 条核心结论

## Prompt 设计

**单部门 Prompt 结构：**

```
你是一位企业经营分析助手。以下是【{部门名称}】{汇报周期}的工作周报数据，
请用专业、简洁的语言生成一段分析摘要（300字以内）。

要求：
1. 先给出一句整体评价
2. 重点说明指标异常（环比下降超过10%需特别指出）
3. 风险问题如有则必须提及
4. 语气客观，不要过度夸奖

数据：
{结构化的部门数据}
```

**汇总 Prompt 结构：**

```
你是一位企业经营分析助手。以下是本周各部门汇报数据的汇总，
请生成一份给管理层的周报摘要（500字以内）。

要求：
1. 开头给出本周整体判断（一句话）
2. 列出表现突出的部门和需要关注的部门，各不超过2个，说明理由
3. 提炼2-3条全公司层面的共性洞察
4. 结尾给出下周重点关注事项

数据：
{所有部门的结构化汇总数据}
```

## 输出：Markdown 报告

每次运行生成一份报告文件，文件名包含日期，存放在 `reports/` 目录下。

报告结构：

```
# 周报分析 · 2025-W23

## 执行摘要
（汇总分析内容）

## 各部门分析

### 销售部
（单部门分析内容）
指标概览表格

### 研发部
（单部门分析内容）
指标概览表格

## 数据异常记录
（校验不通过的文件列表及原因）

## 生成信息
生成时间：xxx  处理文件数：x  调用DeepSeek次数：x
```

## 推送功能（可选，两选一）

**选项 A：钉钉群推送**

通过钉钉自定义机器人的 Webhook，把执行摘要部分（不超过 500 字）推送到指定群。完整报告以文件链接或附件形式附上。

触发条件：报告生成成功后自动推送，如果所有文件都校验失败则推送错误通知。

**选项 B：邮件发送**

通过 SMTP 把完整 Markdown 报告以正文或附件形式发送到指定邮箱列表。邮件标题包含周期和处理部门数。

两个选项都通过配置文件开关控制，不推送时只生成本地文件。

---

## 定时任务

通过 `schedule` 库实现，在 `config.yaml` 里配置触发时间，默认每周一早上 9 点运行。

同时支持命令行手动触发：

```bash
python main.py --run-now         # 立即执行一次
python main.py --date 2025-W22  # 处理指定周期的文件
python main.py --dry-run        # 只提取数据不调用DeepSeek，验证读取逻辑
```

---

## 配置文件

所有可变参数集中在 `config.yaml`，不硬编码在代码里：

```yaml
deepseek:
  api_key: "your-key-here"
  model: "deepseek-chat"

paths:
  excel_dir: "./data/excels"
  report_dir: "./reports"

schedule:
  run_at: "09:00"
  weekday: "monday"

notify:
  enabled: false
  channel: "dingtalk"       # dingtalk / email
  dingtalk_webhook: ""
  email_to: []
  smtp_host: ""
```

## 验收标准

**基础验收：**

> 放 3 个模拟部门的 Excel 文件，跑 `--run-now`
> → reports/ 目录下生成对应的 Markdown 报告，内容包含各部门分析和汇总分析

**异常验收：**

> 放一个格式错误的 Excel 文件
> → 该文件被跳过，错误记录在报告的「数据异常记录」里，其他文件正常处理

**dry-run 验收：**

> 跑 `--dry-run`
> → 控制台打印提取到的结构化数据，不调用 DeepSeek，不生成报告

**推送验收（如果做）：**

> 配置钉钉 Webhook 后跑一次
> → 群里收到执行摘要消息

---

## 不做的事

* 不做 Web 管理界面
* 不做 Excel 模板自动下发
* 不做历史报告对比分析
* 不做多人协作或权限管理
* 不支持非结构化 Excel（格式乱的表不在处理范围内）

## 后续可延伸的方向

这个项目本身就是一个完整的 RPA 雏形。跑通之后有两个方向可以延伸：

一是把它包装成 MCP 工具，让 Claude Desktop 能主动触发分析，比如对话说"帮我分析一下上周的报表"就自动执行；

二是把 Excel 目录替换成数据库查询，从 SQLite 或 MySQL 里实时取数，和项目②③组合，就是一个完整的数据分析 Agent 系统，足够在面试中作为核心项目讲清楚完整的技术链路。
